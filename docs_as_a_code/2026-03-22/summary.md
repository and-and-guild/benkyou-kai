### Cursorコーディングエージェントの導入事例

Money ForwardがCursorのコーディングエージェントをプロダクト・デザイン・QAチームに展開している事例を取り上げた。

背景:

- CursorはUIやセットアップ速度の面でClaude Codeと比べても遜色なく、フロントエンド開発では便利な場面もある
- 個人利用ではトークンのリミットが早くやってくるが、企業規模での利用では規模が大きいほどコストを下げられる可能性がある

注目ポイント:

- QAチームではテストケースとテストスクリプトの作成を自動化し、工数削減を実現
- 導入前はQAエンジニアがプロダクト資料を手作業で読み込み、ユーザーストーリーごとに手動でテストを作成していた
- Claude Codeと同等の性能のモデルを同じだけ使うと、他のツールでは早くリミットに達するという印象がある

参考: [Money Forward × Cursor 事例ブログ](https://cursor.com/ja/blog/money-forward)

### OpenClaw AIエージェント

Amazon LightsailにOpenClawが導入されたことを取り上げた。OpenClawをVPS上で動かすことで、ローカルPCのセキュリティリスクを分離できる仕組みとして注目された。

サービスの概要:

- Amazon LightsailはVPSインスタンスを月額固定料金で提供するサービスで、ConoHa・Xサーバー・さくらサーバーに相当する
- LightsailのインスタンスオプションにOpenClawのセットアップ済み起動オプションが追加された（WordPressセットアップ済みインスタンスと同列）
- OpenClawはブラウザ操作・ローカルファイル操作・ネットワーク通信など幅広い操作が可能であり、VPS環境に閉じ込めることでローカルPCへの影響を防ぐ

コスト構成:

- VPS料金：月額3.5ドルから（月額固定）
- LLM利用料：Amazon BedrockがデフォルトのAIモデルプロバイダーとして事前設定されており、従量課金が発生する
- ローカルLLMに接続する構成であれば電気代のみになる。DeepSeekやGrokのような低コストモデルを使えば月50ドル以内に抑えられる可能性もある

現状の使われ方:

- DeNAが「レモン君」という名のOpenClawエージェントを実験的に稼働させている事例がある
- 現時点では趣味・実験的用途が中心で、夜通し稼働させるような明確なタスクがある場合に活用できる
- 業務での具体的なユースケースはまだ検討段階。QA用途での活用などが候補として挙がった

参考:
- [Introducing OpenClaw on Amazon Lightsail（AWSブログ）](https://aws.amazon.com/jp/blogs/news/introducing-openclaw-on-amazon-lightsail-to-run-your-autonomous-private-ai-agents/)
- [DeNA「レモン君」事例（serverless.co.jp）](https://serverless.co.jp/blog/v0dz9baxe/)

### AIDLC（AI駆動開発ライフサイクル）ワークフロー

AWSが提唱するAI駆動開発ライフサイクル（AIDLC）の概要を調査・紹介した。単なるAIアシスト型ではなく、開発ライフサイクル全体を再設計するアプローチとして位置付けられている。

背景と課題認識:

- 既存のAI活用には「AIアシスト型（コード補完・ドキュメント生成など特定タスクの強化）」と「AI自立型（アプリケーション全体を自動生成）」の2つがあるが、いずれも速度・品質の面で最適とは言えないとされる
- 従来のSDLCは人間主導の長いプロセスを前提に設計されており、AIを単純に後付けで組み込んでも非効率が固定化されるだけとAWSは主張している

コアとなるメンタルモデル:

- 「AIが詳細な作業計画を立て、確認を求め、人間が承認してから実装する」というパターンを全SDLC活動で高速に繰り返す
- AIがルーティンタスクを担い、人間はリアルタイムの問題解決・創造的思考・迅速な意思決定に集中する役割分担

3つのフェーズ:

- **インセプション（Inception）**: ビジネス意図を詳細な要件・ストーリー・作業単位に変換する。チーム全体がAIの質問・提案を検証する「モブエラボレーション」プロセスが特徴
- **コンストラクション（Construction）**: インセプションで検証されたコンテキストをもとに、アーキテクチャ・ドメインモデル・コード・テストを提案する
- **オペレーション（Operation）**: 蓄積されたコンテキストを使い、IaCやデプロイメントを管理する。現時点ではプレースホルダーとして定義されている

新しい用語:

- スプリント → **ボルト（Bolt）**：数時間〜数日という短く集中的な作業サイクル
- エピック → **作業単位（Lesson Unit）** に相当する用語

実装方法:

- GitHubリポジトリのCLAUDE.mdにcore_workflow.mdの内容を置き、rules_detailsディレクトリをローカルに配置することでClaude Codeに適用できる
- AWS固有のサービスへの依存はなく、自分のClaude Codeのプランで試せる
- Audit.mdに全操作ログが自動記録される。questions.mdファイルに「A」「B」などの回答を記入することでAIと対話しながら進める方式

関連リポジトリ:

- [awslabs/aidlc-workflows（公式リポジトリ）](https://github.com/awslabs/aidlc-workflows)
- [core-workflow.md（CLAUDE.mdに配置するワークフロー定義）](https://github.com/awslabs/aidlc-workflows/blob/main/aidlc-rules/aws-aidlc-rules/core-workflow.md?plain=1)
- [gotalab/cc-sdd（類似コンセプトの別実装）](https://github.com/gotalab/cc-sdd)

### AIDLC実践ハンズオン

DioxusとRustを使った勉強会サイト（静的サイト）の構築プロジェクトでAIDLCワークフローを実際に試した。

技術スタック:

- **Dioxus**：Rustのフロントエンドフレームワーク
- テンプレート：Bear Bonesを選択
- スタイリング：Tailwindを使用

実施内容:

- core_workflow.mdをCLAUDE.mdとして配置し、ウェルカムメッセージが表示されることでAIDLCワークフローの動作を確認
- ワークスペース検出 → リバースエンジニアリング → インテント分析 → 要件定義 → エクセキューションプランという流れで進行
- 要件定義では機能要件（FR）と非機能要件（NFR）に番号が振られ、「FR03に対応してください」のように参照しやすい形式で管理された
- 途中でパブリックリポジトリでのホストという要件変更が発生し、ワークフローが要件定義フェーズに差し戻された

技術的な実装内容:

- `build.rs`というビルドスクリプトを生成し、ビルド時にMarkdownファイルからRustソースコードを自動生成する仕組みを採用
- セッション一覧（ホーム画面）とセッション詳細ページを持つ静的サイトが完成
- 最終的に3セッション分のカードが表示され、クリックで詳細ページへ遷移することを確認

実用上の課題:

- コンテキストウィンドウが残り数%まで消費され、完了ギリギリだった
- AIが生成した成果物（アーキテクチャドキュメント・コード構造・APIドキュメントなど）が膨大で、人間側のレビューが追いつかない場面があった
- questions.mdへの回答形式でのやり取りは、人間がどのように操作を求められているかを把握しきれない場面もあった

総評:

- 今回はタスクが単純だったため、Claude Codeのplanモードで同等のことができる可能性もある
- 複雑なプロジェクトでより威力を発揮するか、コンテキストが持つかどうかが今後の検証課題
- 成果物が多くドキュメントが整備されるため、セッション引き継ぎや人間同士の引き継ぎには有用である

### Rustのコンパイル時間とGoとの比較

Rustの2026年ロードマップにコンパイル時間の改善が課題として明記されており、コンパイル時の検証が多い言語の宿命として認識されている。TypeScriptも同様の問題を抱えているが、大企業の大規模投資によって解消されていく傾向があり、Rustも需要増に伴い改善されていくと見込まれる。

GoとRustのパフォーマンス比較（2026年時点の記事）では、バイナリサイズ・メモリ使用量においてRustが優位で、GCを持つGoはスケール時に重くなる傾向があるとされた。ただし実際のプロダクトコードレベルでは差が出にくいという体験談も共有された。

### 依存関係のセキュリティチェック

npmなどのエコシステムにおける依存関係のセキュリティチェックの難しさが議論された。

現状の課題:

- TypeScriptプロジェクトでは利用ライブラリの全依存関係が問題ないかを確認するのは事実上不可能
- 依存関係のネストが深くなりすぎており、全てをたどると1日では終わらない
- 悪意ある振る舞いはソースコードを直接確認するしか検出できないが、それも現実的ではない

現実的な対策:

- 信頼できる提供元のライブラリを使うか、依存関係を絞ることが現実的な対応

参考: [依存関係のセキュリティに関する記事（ITmedia）](https://www.itmedia.co.jp/aiplus/articles/2603/09/news106.html)
