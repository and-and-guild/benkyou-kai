### Claude Code Desktopのアップデート

Xで共有されていたClaude Code Desktopの新機能について議論した。

- デスクトップアプリ内でウェブアプリケーションのプレビューが可能になった
- Claude Code Desktop自体は以前から存在していたが、今回のアップデートでWebアプリをアプリ内で動かし、Claude自身がUIを確認できるようになった
- Cursorなどの他ツールには既にあった機能であり、追いついた形
- 参加者は普段Claude Codeをターミナルで使っているため、デスクトップ版は使っていない
- フロントエンド開発時のレビュー用途（ランニングアプスのプレビュー確認）で特に有用と評価

### サンドボックスとVM環境の検討

Claude Codeの`dangerously skip permission`オプションの使用と安全な実行環境について議論した。

現状の課題:

- 参加者の一人は`settings.json`にallow/denyのパーミッションを書いて業務でも使用中
- サンドボックスモードではプロジェクト外ファイルの読み取り（ライブラリコードの参照など）でパーミッションエラーが発生し、長時間実行時に止まってしまう
- プランモードではサンドボックスモードが最適とベストプラクティスに記載あるが、実行時には制限が厳しい
- 「寝て起きたら早い段階で止まってた」では萎えるという実感

検討した方針:

- サンドボックスの制限を個別に設定するよりも、VMやDockerコンテナで完全隔離した環境を用意する方が効率的
- コンテナ内では自由にやらせ、消されて困るものは置かないという運用
- 成果物はコードだけなのでVM内でも問題なく動作する
- サンドボックスで中途半端にやるよりも、レイヤーでがっつり分けて自由にやらせる方が良いという結論

### Claude Codeの活用パターン

現在の基本的な使い方:

- プランモードで作業を分解し、プランを確認してから実行するのが基本
- 並列実行は「明らかに同時進行できる」場合のみで、基本はシングルタスク
- 社内でも夜間並列実行をやっている人は少数

発展的な使い方（目指している運用）:

1. 大きくやりたいことをClaude Codeに投げて分解させる
2. 分解した各項目をGitHub Issueとして登録
3. 別のセッションを立ち上げ、`gh`コマンドでIssueを取得させる
4. コードベースから事実確認と問題箇所の特定をさせる
5. さらにサブIssueとして細分化（これらは単純作業になる）
6. 日中はプラン作成・サブIssue作成に専念
7. 夜間に単純作業のサブIssueを並列で自動実行

コンテキスト管理のコツ:

- プランをチャット内でやり取りするとコンテキストが汚れるため、一旦Issueに吐き出させてから人間が赤ペン添削する
- 余計なトークン消費を防ぎ、コンテキストの乱れを抑制する
- NotionやGitHub MCPを使って外部ストレージに中間成果物を出し、別セッションから再開する運用を目指す
- ベストプラクティスでも「アウトプットはさせない方がコンテキストが汚れない」と記載あり

### GitHub Organizationとプロジェクト管理の構築

勉強会用のGitHub Organizationを実際に作成した。

組織名の決定プロセス:

- Claudeに候補を提案させた（遊び心のある提案が多かった）
- 「&&hub」「&&club」などの候補を検討
- 多くの名前が既に使われていた
- 最終的に「guild」に決定（少人数で増えない予定なのでギルドが適切）

プロジェクト管理の設定:

- テストリポジトリを作成してプロジェクトボードの動作を確認
- 看板形式（Kanban）のプロジェクトボードを作成
- 当初は「Todo」「In Progress」「In Review」「Done」の4カラムだったが、「In Review」は不要と判断して削除
- ロードマップビューを発見し、期間を可視化できることに喜び（「この本はこの期間から読んでます」が見える）
- ラベルを追加: AI、Rust、データ指向アプリケーション設計、アーキテクチャなど
- メンバーの権限をオーナーに昇格させてラベル作成等の操作を可能にした
- マイルストーン機能は使用経験があるが有用性を感じたことがないため不採用

過去の勉強会記録の整理:

- 2024年5月末頃から勉強会を開始
- マイクロサービスの本は2024年10〜11月頃に取り組んでいた
- Rustは2025年1月頃から開始
- ロードマップビューで過去の活動期間を登録

### 二重リポジトリ戦略

プライベートとパブリックの2つのリポジトリを使い分ける戦略を検討・採用した。

背景と課題:

- 勉強会の議事録を積極的に公開していきたい
- 万が一含まれてしまった場合に備えてセーフティネットを設けたい
- 技術的に興味深いアプローチで解決したい（運用でカバーだけでは面白くない）
- 公開用に記事化して社内外にアウトプットしたい

検討した代替案:

1. **SOPS（Secrets OPerationS）による暗号化**: KMSやageなどの鍵管理と組み合わせてGitリポジトリ内のファイルを暗号化する手法。会社でKubernetesのシークレット管理に使用している実績あり。ローカルで秘密鍵を使って復号する運用。ただし非構造化データ（議事録）の暗号化は取り回しが悪く、MCPから直接読むと暗号化されたままになる課題がある
2. **サブモジュールで機密部分を分離**: プライベートリポジトリをサブモジュールとして埋め込む
3. **運用でカバー**: 公開を前提に意識して会話する（コスト最低だが技術的に面白くない）
4. **GitHub Secret Scanning**: クレデンシャル漏洩検知ツールだが、今回の非構造化データの問題とは異なる

採用した方針:

1. プライベートリポジトリで全作業（PR・レビュー）を実施
2. プライベートのメインブランチへのマージ時に、GitHub Actionsでパブリックリポジトリへ自動ミラーリング
3. パブリックリポジトリでトークンを発行し、プライベート側のワークフローからプッシュする
4. PRはプライベート側のみで実施し、パブリックにはメインブランチのみをミラーリング
5. 念のため、Copilotのレビューで公開前のセーフティチェックを自動化することも検討

この戦略自体が記事のネタになりうるという認識もあり、二重リポジトリ戦略を技術記事として発信する計画も出た。

### 議事録の管理方法

議事録のワークフローと構造について詳細に検討した。

入力ソース:

- Notion議事録の原文（音声書き起こし）
- Notionの要約機能による要約

課題:

- 原文は音声書き起こしのため「あの」「はい」などのフィラーが混在し整理が必要
- 原文をそのままAIに要約させると品質が安定しない
- Notionの要約だけでも十分とは限らない

採用したワークフロー:

1. Notionは原文の記録に特化する
2. 原文とNotionの要約の両方を入力として、別のエージェントが固定フォーマットで要約を生成する
3. 公開するのは要約のみ。原文は非公開だがプライベートリポジトリに保管（後で何かに使える可能性）

ディレクトリ構造の検討:

- 分野ごとにディレクトリを切る案 vs 日付ごとにフラットに並べる案
- 最終的に日付ディレクトリ（例: `2026-02-22/`）の下に原文（`original.txt`）とサマリー（`summary.md`）を置く構造を採用
- 分野ごとの分類は後から必要になったら再構成すればよいという判断
- ファイル名で内容がわかるようにしたいという要望もあったが、一旦日付ベースで開始

音声書き起こしツールの話題:

- AquaVoiceやVoiceInkなどのツールに言及
- AquaVoiceはキーボード入力の代替（音声入力メソッド）であり、長時間の議事録向けではない認識
- Google MeetのGemini機能で同様のことができる可能性もあるが、無料プランでは怪しい
- 結局、議事録の書き起こしに無料で良いものはなさそう

### コスト感覚と組織の予算

雑談の中で出た組織のコスト感覚:

- AWSのコストが5万円上がっただけで会議で10分議論になる文化
- 部署間の予算管理が細かく、所属と実際の業務部署が異なるケースもある
- 個人PCの環境改善（モニター・キーボード等）は全社一律の制約があり、部署判断で柔軟に対応しにくい
- エンジニアのツール投資（月100〜200ドル）は人件費と比べれば微々たるものだが、部署予算ではシビア
