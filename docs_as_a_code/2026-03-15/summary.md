### 記事レビュー:「シンプルな設計をするエンジニアは出世しない」

エンジニアリングにおけるシンプルさと複雑さの評価構造に関する英語記事（[Nobody Gets Promoted for Simplicity](https://terriblesoftware.org/2026/03/03/nobody-gets-promoted-for-simplicity/)）をレビューした。複雑なシステムを構築したエンジニアの方が昇進審査で評価されやすく、シンプルに実装したエンジニアの仕事は「見えなくなってしまう」という構造的な問題を指摘している。

記事の要旨:

- エンジニアAはシンプルな実装を選び、素早く出荷して次に移る。エンジニアBは新しいミドルウェアの導入、イベント駆動アーキテクチャの構築、設定フレームワークの構成など複雑なアプローチを取る。昇進審査ではBの方が「語れる物語」を持つため有利になる
- 面接のシステム設計でもシンプルな解決策は「十分に面白くない」と見なされ、シャーディングやキュー、サービス分割を追加することを求められる。候補者は「複雑さが人を感心させる」という教訓を現場に持ち込んでしまう
- 設計レビューでも「将来に備えておくべきでは」という声に押されて、まだ必要のない抽象化層や柔軟性を追加してしまう傾向がある
- シニアへの実際の道は、ツールやパターンをより多く学ぶことではなく、それらを「いつ使わないか」を学ぶことである

記事が提案する対策:

- シンプルな仕事の背後にある判断を可視化する。「Xを実装した」ではなく「3つのアプローチを評価し、素直な実装が全ての要件を満たすと判断して2日で出荷した」と記述する
- 設計レビューで複雑さを求められた際は、「後で必要になったときに追加するためのコスト」と「今追加することのコスト」を示して反論する
- リーダーは「出荷できる最もシンプルなバージョンは何か」を問い、複雑さに立証責任を負わせるべきである
- コードを削除したエンジニアや「これはまだ必要ない」と正しく判断した人を公に称えるべきである

議論で出た論点:

- ライブラリのインターフェース設計では、様々なパターンに対応できる柔軟性を追求すると楽しいが、実体がどんどん複雑になっていくというジレンマがある
- ジュニアエンジニアがシンプルな設計を提案しても「それしか知らないのか」と思われるリスクがあり、シンプル側に倒す判断には発言力が必要になる。複雑な設計はコードで表現できるため説得しやすいが、シンプルな設計は「作らなかった理由」を言葉で説明する必要があり、説得のハードルが高い
- 分散DBのようにマネージドサービスとして利用する場合は、最初からスケールするものを選んでもコードの複雑さは増さないため許容される。特にデータベースは後から移行するのがほぼ不可能であるため、最初の選定が重要である
- 面接で複雑なアーキテクチャを語れる方が「すごそう」に見えるのは事実だが、それを現場に持ち込んで最初からオーバーエンジニアリングするのは危険である

### Agent Workspaceの紹介

コンテナ環境でClaude Codeを実行できるOSSツール「[Agent Workspace](https://github.com/hiragram/agent-workspace)」を紹介した。3月10日頃にリリースされたもので、コンテナ内でClaudeを動かすために必要なスクリプトが一式揃っている。

注目ポイント:

- 対象リポジトリのコードをメインブランチから取得し、ワークツリーとして分岐させることで環境を分離できる
- `dangerously--skip-permissions`オプションを使って自動実行が可能
- 複数のプルリクエストを並列に処理できるため、効率的に作業を進められる

### Claude Codeのループモードの活用

Claude Codeのループコマンド（`loop`）の活用方法について議論した。実質的にはcronのように定期実行させる仕組みで、PRレビューの自動化に大きな可能性がある。

運用イメージ:

- PRを作成した後、ループモードで「このPRにレビューコメントがあれば、その内容を反映してコミット・プッシュする」という指示を1分間隔で実行させる
- レビュアーは自然言語でコメントを書くだけで、コードが自動的に修正されていく
- 最大3日間動作し、その後自動的に終了する
- 3日を超える長期運用にはcronジョブの方が適している

発展的な使い方:

- 仕様をチームで決めた後、Claudeに初期実装をさせてPRを作成し、チーム全員でレビューする運用。PR作者が特定の個人に帰属しなくなる
- クラウド環境で作業をキックしてPRを作成し、ループモードでレビュー対応を自動化する流れ

### mirror.mdスキルの作成

mirror.mdの生成プロセスをClaude Codeのスキルとして定義した。スキルクリエイター（スキルを作成するためのスキル）を導入し、それを使ってmirror-creatorスキルを作成した。

背景:

- これまでmirror.mdの生成はClaude Codeへの手動プロンプトで行っていたが、毎回プロンプトを思い出す必要があった
- スキル化することで、入力ファイル（original.txtとsummary.md）の指定、フォーマットルール、情報の取捨選択基準を再利用可能な形で定義できるようになった
- 日付ディレクトリが指定されない場合にmirror.mdが存在しないディレクトリを探して提案する機能も含まれている

ループモードとの連携:

- スキル作成のPR自体もループモードでレビュー対応を自動化する体験を実施した
- レビューコメントを書くと自動的にコードが修正されてコミットされる流れを確認できた
