### 技術トレンド収集システムの構築

YouTubeチャンネル「Tech World」の動画を参考に、技術情報を自動収集するシステムを開発・紹介した。

- AWS、Rust、Claudeに関する技術トピックを自動収集するClaude Codeのスキルとして実装
- 各分野ごとにソースを設定（AWSはドキュメントMCP、Rustは公式コミュニティなど）し、直近3日間のトレンドを取得
- 取得結果をJSON形式で出力し、静的なフロントエンドサイトでレンダリングして閲覧できるようにした
- バックエンドは不要で、GitHub Pagesなどで完全に静的ホスティングが可能
- 日常のルーティンに組み込むことで、依存ライブラリの破壊的変更やリリース情報を即座に把握できる
- 動画内では、収集したトレンドから気になる記事を深掘りしてコンテンツのネタにするサイクルが紹介されていた
- CLAUDE.mdでセキュリティ寄りなど自分の関心領域を指定することで、取得する情報の質を高められる
- 現時点では手動実行だが、GitHub ActionsやMacのcronジョブで自動化する計画もある
- 公開しても独自性のある情報ではないため問題なく、「自分はこれで情報収集しています」というスタンスで共有可能

### リポジトリミラーリングワークフローの実装

前回検討した二重リポジトリ戦略を実際に実装した。プライベートリポジトリからパブリックリポジトリへのコンテンツ自動同期をGitHub Actionsで構築。

トリガー方式の検討:

- `workflow_dispatch`（手動実行）と`on push`（プッシュ時自動実行）の両方を比較検討
- `workflow_dispatch`では差分の特定が難しく、引数でファイルパスを指定する必要がある
- `on push`（メインブランチへのプッシュ時）の方が、PRの差分ファイルを自動で取得でき扱いやすいと判断
- 最終的にメインブランチへのプッシュ時に`docs_as_a_code/**/*summary.md`の変更を検知してトリガーする方式を採用

差分検出の技術的検討:

- `fetch-depth`の設定について議論。デフォルトの1だと親コミットが取得できずdiffが取れないため、2に設定
- リベースマージで3コミット以上ある場合の問題点も検討したが、`github.event.before`と`github.event.after`のSHAを使うことで対応
- 変更されたファイルを抽出し、チェックアウトしたパブリックリポジトリに同じファイルパスでコピーする方式

トークンとセキュリティ:

- パーソナルアクセストークン（PAT）のクラシックとファイングレインドを比較
- クラシックPATはリポジトリ範囲を限定できない問題がある
- ファイングレインドPATを採用し、対象リポジトリを限定、コンテンツの読み書き権限のみを付与
- 理想的にはGitHub Appトークン（`create-github-app-token`アクション）への移行が望ましいが、一旦ファイングレインドPATで運用開始
- PATにはユーザー紐づきの問題（退職時にワークフローが止まる）があることを認識

実装結果:

- ワークフローが正常に動作し、プライベートリポジトリのPRマージ時にパブリックリポジトリへの自動同期を確認
- コミットメッセージは「Publish from private repo」で統一
- コミッターはBotアカウントとして表示される
- 将来的にはパブリック側でさらに静的サイト生成のワークフローを連鎖させることも検討

### Zedエディタの試用レポート

参加者がRust製の新しいエディタ「Zed」を試用した感想を共有した。

良い点:

- 起動速度が全体的に速い
- デフォルトのUIがVS Codeよりシンプルで好み
- GitHub Copilotとの連携が可能（プロプランでなくてもCopilot経由の補完は使える）

課題:

- Vimモードのデフォルト設定でクリップボードとヤンク領域が分離されている（コピー→削除→ペーストで削除内容が貼り付けられる問題）
- Makefileでタブ文字が自動挿入されず、スペースが入ってしまう（Makefileはタブ文字必須）
- エディタ予測機能（Edit Predictions）は有料プランが必要な可能性あり（2000回制限の報告）
- プラグインで解決可能だが、「デフォルトの思想」としてどこまで手を入れるかという議論に

VS Codeとの比較:

- Zedは「余計なことはしない」思想だが、一方でクリップボード分離のように「残しておく」方針もあり、一貫性に疑問
- CursorのCompanion機能（ウィンドウ内プレビュー、DevTools連携）のような便利機能はまだない
- 現時点ではCursorから移行する決め手には欠ける

### CursorとClaude Codeのコスト・レート比較

- Cursorはトークン消費が激しく、3日で月間上限に到達してしまった
- 1日あたり約20ドル分の使用量（高額なモデルを使っていた可能性）
- Claude Code（MAX 100ドルプラン）の方がレート制限に余裕がある印象
- MCPのスキーマ情報がトークンを消費している可能性や、不要なMCPが有効になっている可能性を指摘
- GitHub MCPを廃止してghコマンドに切り替えることでトークン消費を削減するアプローチを実践中
- MCPよりも既存のAPIやCLIコマンドを使う方が、余計なツール読み込みがなく効率的

### freee MCPの紹介

freeeのAPIをMCP化したオープンソースプロジェクトが公開されている話題を共有した。

- 確定申告をシェル（CLI）から実行可能にするヘッドレスアプリケーション
- GitHubで213スター獲得しており注目度が高い
- freeeの社員が個人プロジェクトとして開発し、後に会社に戻して公開した形
- コミット履歴を見るとほぼClaude（Bot）がコミットしており、AIで開発されたと推測される
- パブリックAPIを叩くだけのツールなので、API仕様自体も公開情報
- まだアーリーアダプター層が中心だが、こうしたヘッドレスツールが広まれば世界観が変わる可能性がある

### MCP活用の現状と方針

MCPの使いどころについて議論した。

- GitHub MCPはghコマンドで代替可能で、むしろCLIの方がトークン消費が少なく効率的
- Notion MCPは引き続き有用
- 既存のAPIがあるサービスについては、MCPよりもAPIを直接叩くコマンドを用意する方が良いという方向性
- 社内ブログ記事「WebAPIをMCP化してエージェントレディにした話」を確認し、OpenAPIスペックからMCPサーバーへの変換が可能であることを把握
- プロダクトでMCPサーバーを開発する機会はまだ少ないが、将来的に役立つ知識として蓄積

### 議事録ファイル名の整理

ミラーリングの実装過程で、パブリックに公開されるファイル名について議論した。

- 当初`summary.md`をミラー対象としていたが、ワークフローのトリガー条件で`mirror.md`というファイル名に変更
- パブリック側に公開されるファイル名は将来変更しにくいため、慎重に検討
- 最終的に`summary.md`をトリガー対象とし、同じファイルパスでパブリックにミラーする方式に落ち着いた
